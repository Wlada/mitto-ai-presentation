# Presentation Overview

> One-page mental model of the entire presentation. Read first.

---

## The whole thing in one sentence

A 30-minute talk that demonstrates Claude Code's full-stack feature delivery
workflow on a presentation app that itself renders the slides.

## Audience

Mixed technical team — developers, QA, tech leads. Some familiar with AI
tools, most not familiar with Claude Code specifically.

## Goals

1. **Evaluation** — give them enough to decide whether to adopt Claude Code
2. **Education** — show what's realistic, including limits

We are not selling. We are not promoting autonomy. We are showing a
controlled engineering process.

## The central message

> The value isn't agent autonomy. It's a faster, controlled cycle where the
> developer stays the owner.

## Time budget

| Block | Minutes | Content |
|-------|---------|---------|
| Intro + concept | 0–5 | Slides 1-3 |
| Setup demo task | 5–9 | Slides 4-6, repo tour |
| Live demo | 9–18 | Brainstorm → plan → dispatch → cut |
| Pre-built result | 18–25 | Slide 7 (Results numbers), diff, code reviewer, tests, coverage |
| Limits + Q&A | 25–30 | Slides 8-10 |

The deck is **10 slides** total. Slide 7 ("Results") shows live test and
coverage numbers read from `apps/web/src/app/slides/slide-07-results/results.data.ts`,
which is regenerated by `npm run results:refresh` (run as a pre-show step).

## The "meta" trick

The application that renders the slides IS the application the agent works on.
Audience sees this when:
- Slide 4 mentions "the slides you're seeing right now"
- The browser address bar shows `presentation.vladimirbujanovic.com/slides/N`
- The same `/feedback` route they see in slides is **already live** at the
  same URL — they can pull it up on their phones during the talk
- The deployed branch (`demo-finished`) is the finished result the
  workflow produces, so the audience never has to wait for a `git
  checkout` mid-demo

This is one trick that pays off three times: it explains the architecture,
gives the live demo a coherent target, and makes the "leave a question"
button at the end real.

## The workflow

Every non-trivial change in this repo flows through a six-step cycle,
defined canonically in `CLAUDE.md`:

1. **Brainstorm** — `/superpowers:brainstorming`
2. **Plan** — `/superpowers:writing-plans` (writes to `docs/plans/`)
3. **Execute** — `/superpowers:subagent-driven-development` (parallel) or
   the main agent (single-layer)
4. **Simplify** — `/simplify` skill
5. **Review** — `code-reviewer` subagent
6. **Document** — update `CLAUDE.md`, `README.md`, `docs/`

The Audience Q&A feature on `demo-finished` was built end-to-end with this
workflow. The reproducible recipe — Prompts 1 through 7 — lives in
`04-prepared-prompts.md`.

## Two branches, one URL

- `main` — slides + scaffold + docs. **Local working copy stays here for
  the entire talk.** Claude Code runs the live agent flow against this
  branch — the agent sees a fresh repo with no Q&A feature, so the
  brainstorm questions are meaningful.
- `demo-finished` — `main` + the full Q&A feature with tests. **This is
  the branch deployed at <https://presentation.vladimirbujanovic.com>.**
  The audience sees the finished state on the live URL throughout the
  talk; no `git checkout` happens mid-demo.

When the live agent flow ends (around minute 18), you just shift focus
from terminal to the browser tab on the live URL — already showing the
finished result — instead of swapping local branches.

## Terminals

- Terminal 1: `claude` (running Claude Code on local `main`)
- Terminal 2 (optional): `npm run dev` if you want a local preview tab in
  addition to the live URL on the projector. Skip if you don't.

## Two screens (recommended)

- Main screen: browser + terminal (audience sees this)
- Side screen: cheatsheet + prepared prompts (only you see this)

If you only have one screen, print the cheatsheet on paper.

## When to start preparing

The week-by-week plan is in `06-rehearsal-checklist.md`. The short version:

- 4 weeks out: this doc exists, you're aligned
- 3 weeks out: slides have real content, demo polished
- 2 weeks out: rehearsals start; first live run by Wednesday
- 1 week out: backups recorded, rehearsal video reviewed
- Day of: pre-show checklist, deep breath, you've got this
